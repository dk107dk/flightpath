Why Am I Seeing These Results?

The results of your run are determined by the settings and values below. The middle column applies to this run. These settings will help you better understand why you see what you see.

If you are seeing mainly exceptions or logging from a one-off run try setting the error policy to include print and not raise. You can do this in the config/config.ini using the Config Panel. See the button at the bottom left-hand of the window.

If you still do not understand your results, look to the function and qualifiers definitions in the documentation window at the top left to check your match components. CsvPath Language is functions-based and the qualifiers have a major impact on how the functions evaluate.

Basic Info

Field Value this run What does it mean?
Identity {{data.identity}} The identity of a csvpath is shown in validation errors and other places where it may help you understand your runs. You can set the identity with a metadata field in an external comment. See the Keywords and Symbols section in the documentation window on the right for more details.
Data file {{data.filename}} This is the source of the data your run processed.
Scanned lines {{data.scan_count}} Your scanning instructions in the first bracketed part of your csvpath resulted in this many lines being considered.
Matched lines {{data.match_count}} This is the number of lines that matched your match components in the second bracketed part of your csvpath. There are many reasons why you may not see the number of matches you expected. In general, either the configurations on this page or your match components -- or both -- are the reason.
Blank lines skipped {{data.skip_blank_lines}} The CsvPath Framework skips blank lines by default. This is a programmatically settable parameter. At this time FlightPath does not offer a configuration allowing you to set this behavior within the UI.
Delimiter {{data.delimiter}} Your delimiter is the character that separates data fields. For one-off runs in FlightPath you can set the delimiter using the test-delimiter metadata field. You can learn how to do that in the documentation window to the right under Keywords and Symbols. The default is the comma.
Quotechar {{data.quotechar}} The quotechar is the character that wraps data fields containing controlled characters. For one-off runs in FlightPath you can set the quotechar using the test-quotechar metadata field. Learn how to set that in the documentation window to the right under Keywords and Symbols. The default is the double-quotes.
Configured to collect lines {{data.collecting}} One-off runs in FlightPath are always configured to collect lines. This is because they are run using the collect() method on the CsvPath class. If you ran a named-paths group using one of the fast-forward methods your run would not be configured to collect lines because the fast-forward methods drop matched and unmatched lines, resulting in metadata, variables, and errors only.
Error format {{data.error_format}}

Validation errors are given in a format you control in config/config.ini. You can change the format in the Config Panel. See the Open config button at the bottom left.

An error that looks like:

2025-05-09 13h37m53s-073334:test.csv:2::hello world data model:add[1]: could not convert string to float: 'fix me'

Comes from this format:

{time}:{file}:{line}:{paths}:{instance}:{chain}: {message}

It gives you:

  • - The timestamp of the error
  • - The data file
  • - The line number where the error occurred
  • - The named-paths group name
  • - The csvpath's identity
  • - A 0-based hierarchical "chain" pointing to the source match component
  • - An explanatory message

You can configure a simpler format if you don't need all of this information, as is often the case during development.

Detailed Run Configuration

Field Value this run What does it mean?
Logging level {{data.csvpath_log_level}} The log level is set in config/config.ini. It must be one of ERROR, WARN, INFO, or DEBUG. You can set the log level in the FlightPath Config Panel. See the button at the lower left to open the panel.
Errors policy {{data.csvpath_errors_policy}}

The errors policy determines how CsvPath Framework behaves when it sees an error. You change the settings in the FlightPath Config Panel. See the button at the lower left to open the panel.

The errors policy can have a huge effect on what you see. If your run has an error and your policy is to raise exceptions on errors, your run will fail and you will only see the log trail. Likewise, if your policy is to stop the run on errors your run will terminate at the first error. For development you probably want to print errors rather than raise exceptions to them.

There are two ways to set the error policy. First, the config file. But overriding that, there is a mode that allows you to set the behavior on a csvpath-by-csvpath, run-by-run basis. The mode is validation-mode. See the documentation window on the left-hand side for more on the modes, including validation-mode.

Stop on validation errors {{data.stop_on_validation_errors}}

If you configure to stop on validation error, your run terminates at the first error.

You can set this policy in two ways. First, using the main config file at config/config.ini. Second, and overriding the config file, you can set validation-mode to include stop or no-stop. See the documentation window on the left-hand side for more on the modes.

Fail on validation errors {{data.fail_on_validation_errors}}

CsvPath Framework can set a run to invalid if it sees an error. You can also do this in your match component rules using fail(). Failing a run is just a way to signal to yourself or a data consumer that the data coming from a run is considered no good.

You can set this policy in two ways. First, using the main config file at config/config.ini. Second, and overriding the config file, you can set validation-mode to include fail or no-fail. See the documentation window on the left-hand side for more on the modes.

Print validation errors {{data.print_validation_errors}}

Printing validation errors is the best way to see them and take action.

You can set this policy in two ways. First, using the main config file at config/config.ini. Second, and overriding the config file, you can set validation-mode to include print or no-print. See the documentation window on the left-hand side for more on the modes.

Log errors {{data.log_validation_errors}} CsvPath Framework logs errors by default.
Raise exception on error {{data.raise_validation_errors}} Raising validation errors is valuable for some purposes, especially in development. However, it tends to minimize the information you receive from a run. It is generally better print errors when you are working in development.
Match on error {{data.match_validation_errors}} You can choose to match lines with errors, rather than matching lines without errors, the default. This means that calling collect() or next() will return only the bad lines. Match on error is a mode called match-mode. You can learn more about the modes in the top right-hand documentation window.
Collect validation errors {{data.collect_validation_errors}}

You can choose to not capture errors. If you do not, you will not see any data in the Errors tab. Generally, it is better to collect errors.

You can set this policy in two ways. First, using the main config file at config/config.ini. Second, and overriding the config file, you can set validation-mode to include collect or no-collect. See the documentation window on the left-hand side for more on the modes.

Explain mode {{data.explain}} When you set Explain Mode CsvPath Framework writes out a line-by-line, match-component-by-match-component explanation for why a line matched or did not match. You can learn more about the modes in the documentation window at the top-right.
Collect when not matched {{data.collect_when_not_matched}} This setting determines if collect() results in a set of unmatched lines. It is not used in one-off runs in FlightPath. When you run a named-paths group any csvpath that uses this mode results in an unmatched.csv file with the other run results files. You can learn more about the modes in the documentation window at the top-right.
Stopped {{data.stopped}} The stopped property of the run tells if the run completed or was stopped in the middle. A run can be stopped by an error or by the csvpath writer using stop()
Valid {{data.is_valid}} The is_valid property of the run tells if the run was failed in the middle. A run can be failed by an error or by the csvpath writer using fail()
Logic Mode {{data.logic_mode}}

This mode determines if the CsvPath Framework uses AND or OR logic in the evaluation of match components. This means that you can write expressions where if any one match component evaluates to True the line is matched. By default, all match components must evaluate to True for a line to be matched.

With Logic Mode there are three ways to handle match components in lines:

  • - All participating match components must match
  • - Any one participating match component must match
  • - Match components not participating in the evaluation may match or not in order to trigger side-effects

The top two options are mutually exclusive. AND or OR logic can be mixed, though with less conciseness, using boolean functions.

You can learn more about the modes in the documentation window at the top-right.

Print Mode {{data.print_mode}}

This mode determines if printout lines go to the default printer or not. If the default printer is not used, and no other printer is available, you will not see printouts. This mode is generally not useful in one-off runs during FlightPath csvpath development. It is mainly useful for named-paths group runs, triggered in FlightPath or by automation.

You can learn more about the modes in the documentation window at the top-right.

Printers {{data.printers}} CsvPath Framework can use any number of Printer objects. In practice, usually all print streams go through one default printer. That one printer object can take any number of print streams so that you can separate different kinds of output.
Run Mode {{data.will_run}}

Run Mode is used to temporarily disable a csvpath. It is mainly relevant to named-paths groups, where can be helpful to run the group without certain csvpaths participating in a particular run. In a one-off run during development in FlightPath it is less useful.

You can learn more about the modes in the documentation window at the top-right.

Unmatched Mode {{data.unmatched_mode}}

This mode determines if unmatched lines are retained or dropped. Within a named-paths group run, if unmatched lines are retained they are captured to an unmatched.csv file.

Unmatched Mode is not used in one-off runs during FlightPath development. FlightPath does not capture lines, matched or unmatched, to a file. However, using FlightPath you can do an ad hoc named-paths group run, and in that case you would see the unmatched lines file.

You can learn more about the modes in the documentation window at the top-right.

Source Mode {{data.data_from_preceding}}

Source Mode is how you can pull results data from one named-paths group member to the next in the same run. When Source Mode is set to preceding the first csvpath in the group uses the source data provided for the run. Every subsequent csvpath that has Source Mode preceding takes the preceding csvpath's data.csv output as its input.

Note that Source Mode is only relevant for named-paths group runs using the collect_paths, fast_forward_paths, or next_paths methods. The analogous breadth-first methods, collect_by_line, fast_forward_by_line, and next_by_line by their nature feed the output of one csvpath into the next csvpath in the group.

You can learn more about the modes in the documentation window at the top-right.
