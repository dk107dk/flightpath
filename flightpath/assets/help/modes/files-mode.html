Mode: files-mode

The files-mode setting tells CsvPath Framework that the run instance manifest and the csvpath's manifest should show that files were created as expected, or not. Files mode is only used with named-paths group runs. Files are not generated in one-off validations.

There are various reasons why printouts.txt, data.csv and unmatched.csv might not be generated. For e.g., if we expect no validation output from user-created print() statements or built-in validation error messages, we might set files-mode; to no-printouts. If a validation error was then printed we would be alerted in the metadata.

In another example, if we set unmatched-mode to no-keep (the default) and files-mode to unmatched we have a conflict that we'll be alerted to in the metadata. Similarly, if we set files-mode to data and then run fast_forward_paths() we will not get data.csv files and the metadata will alert us to the mismatch.

errors.json, vars.json, meta.json, and manifest.json are always generated, regardless of files-mode. When you set files-mode to all CsvPath Framework will double-check that meta, vars, errors files were correctly created, but that part of its checking is largely superfluous.

all All file types are expected to be generated
data / no-data Determines if the data.csv file is expected
unmatched / no-unmatched Determines if the unmatched.csv file is expected
printouts / no-printouts Determines if we expect anything to be sent to the Printer instances using print()

Usage

The possible settings can be combined. This combination requires a data.csv and an unmatched.csv to both have been created. ~ files-mode:data, unmatched ~ $[*][ yes() ]
If there are no errors and no other printouts we would not expect a printouts file. ~ files-mode:no-printouts ~ $[*][ yes() ]
