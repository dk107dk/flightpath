Mode: logic-mode

logic-mode controls the relationship between match components during line-matching. For each line of a data file the match components together determine if the line is considered to have matched. A match component is each unit of logic. For example @a == 1. In this case, the attribute a is equal to the integer 1. If we put two of these units of logic into a csvpath there are two match components. logic-mode tells us if we are ANDing the two components or ORing them.

Csvpaths are essentially predicate logic rules made of simple components. If we AND the components we are saying that all components must be true for a line to match. If we OR the components we consider the line to match if any one of the components is true.

The values for logic-mode are AND and OR.

Keep in mind that when we say a line matches we are only saying that all the match components taken together say the line should be kept. We are not saying if the line is valid or invalid. You can keep matching or unmatched lines. It is up to the csvpath writer to determine if a matched line is valid or invalid. This stance is different from validation languages like XSD, DTD or SQL's DDL, where a data passes a schema or not. CsvPath Language is more like Schematron or SQL's DML (i.e. queries, not schemas).

Usage

AND means all match components must agree on true for a line to match. This csvpath will match all lines because its two match components are always true.

~ logic-mode:AND ~ $[*][ yes() 1 == 1 ]

OR means a match component be true for a line to match. This csvpath will never match because its two match components are never true.

~ logic-mode:OR ~ $[*][ no() 1 == 2 ]

This csvpath will always match because one of its two match components is always true.

~ logic-mode:OR ~ $[*][ yes() 1 == 2 ]
